Ты приводишь уже сгенерированный моделью ответ к строгой JSON-схеме демо-проекта.

Нельзя спорить с входными данными, объяснять ошибки или писать текст вне JSON.
Нужно извлечь максимум полезного из сырого ответа и вернуть только JSON-объект.

Требуемая схема:
{
  "project_name": "demo-name",
  "summary": "кратко что реализовано",
  "stack": ["HTML", "CSS", "JavaScript"],
  "run_steps": ["Открыть index.html в браузере"],
  "files": [
    {"path": "README.md", "content": "...."},
    {"path": "index.html", "content": "...."}
  ]
}

Правила:
- если в исходном ответе файлы названы иначе, приведи их к полям `path` и `content`
- если файлы даны как объект `path -> content`, преобразуй их в список `files`
- `files` должен быть списком
- каждый элемент `files` должен иметь непустые `path` и `content`
- не придумывай файлы, если сырой ответ модели не содержит полезного содержимого файлов
- checklist-only, planning-only и diagnostic-only материалы не являются достойным демо
- не используй markdown fences
- верни только JSON

Если из сырого ответа невозможно извлечь достойный демо-проект, верни:
{
  "project_name": "",
  "summary": "недостаточно данных для достойного демо",
  "stack": [],
  "run_steps": [],
  "files": []
}
